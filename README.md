Высокопроизводительный мультитредовый бенчмарк, основанный на парадоксе бесконечности (Теорема о бесконечных обезьянах). 

Программа генерирует случайные строки или перебирает все возможные комбинации, пока не будет найдено точное совпадение с эталонным текстом. Основная цель — измерение скорости генерации (Operations Per Second) и проверка вычислительных мощностей системы на предельных нагрузках, в зависимости от размерности алфавита (локализации).

---

## 🧠 Математическая суть и Парадокс

Если обезьяна будет случайно нажимать на клавиши печатной машинки бесконечно долго, она почти наверняка напечатает любой заданный текст (например, «Войну и мир»). 

Вероятность случайного совпадения строки длины $L$ из алфавита размером $N$ вычисляется как:
$$P = \frac{1}{N^L}$$

**В чем суть бенчмарка?** 
Размер алфавита $N$ кардинально влияет на сложность. Для английского алфавита $N$ мало, для китайского — огромно. Этот бенчмарк позволяет измерить, как меняется скорость перебора в секунду (Ops/sec) в зависимости от "локали" (размера алфавита), и как железо справляется с энтропией.

---

## ✨ Ключевые особенности

- **Два режима работы:**
  - `random`: Чистый Монте-Карло. Потоки генерируют абсолютно случайные строки заданной длины. Возможны дубликаты, но нулевой оверхед на синхронизацию состояний.
  - `brute`: Детерминированный перебор. Потоки делят пространство вариантов (N-ричную систему счисления) и последовательно перебирают комбинации от `ааа...` до `яяя...`. Гарантированное нахождение за конечное время.
- **Zero-Allocation Hot Path:** В режиме `random` строка генерируется и проверяется прямо на стеке (Stack Allocation). Отсутствие выделения памяти в Heap (ОЗУ) позволяет достичь миллионов операций в секунду на одно ядро.
- **Нативная поддержка UTF-8:** Программа работает не с байтами, а с реальными Unicode-символами. Кириллица, иероглифы, эмодзи — всё корректно обрабатывается и проверяется.
- **Динамический алфавит:** Программа автоматически извлекает уникальные символы из эталонного файла, формируя алфавит. Это идеально подходит для тестирования разных локалей.
- **Кроссплатформенность:** Единая кодовая база C++17 работает как на x86_64 (с инструкциями AVX2/AVX-512), так и на Apple Silicon ARM64 (NEON).

## 🛠️ Компиляция

Для получения честных результатов необходима компиляция с максимальной оптимизацией под конкретное железо (`-march=native`).

### macOS (Apple Silicon M1/M2/M3/M4) - Clang
```bash
clang++ -O3 -march=native -std=c++17 -pthread monkey_benchmark.cpp -o monkey_bench
```

### Linux (x86_64 / Intel / AMD) - GCC
```bash
g++ -O3 -march=native -std=c++17 -pthread monkey_benchmark.cpp -o monkey_bench
```

### Windows (MSVC) - Developer Command Prompt
```cmd
cl /O2 /arch:AVX2 /std:c++17 monkey_benchmark.cpp
```

## 🚀 Запуск

```bash
./monkey_bench <reference_file> <mode>
```

- `<reference_file>`: Путь к текстовому файлу с эталонной строкой.
- `<mode>`: `random` или `brute`.

## 🌍 Бенчмарк локалей (Как тестировать)

Главная фишка этого проекта — возможность сравнить, как система справляется с разной энтропией. Создайте файлы с одинаковым количеством символов, но из разных языков, и запустите бенчмарк.

### Подготовка файлов локалей

Создайте 4 файла по 10 символов в каждом:

**1. `en.txt` (English / Latin)**
```text
HelloWorld
```
*Размер алфавита: 7 уникальных символов. Вероятность: $1 / 7^{10} \approx 1 / 282M$*

**2. `ru.txt` (Russian / Cyrillic)**
```text
ПриветМир
```
*Размер алфавита: 9 уникальных символов. Вероятность: $1 / 9^{10} \approx 1 / 3.4B$*

**3. `jp.txt` (Japanese / Kanji)**
```text
日本語テスト
```
*Размер алфавита: 5 уникальных символов (но многобайтовых).*

**4. `emoji.txt` (Emojis)**
```text
🚀🌍💡🔥💻
```
*Размер алфавита: 5 уникальных символов (4 байта на символ).*

### Проведение тестов

Запустите программу в режиме `random` для каждого файла на 10-20 секунд (прерывайте `Ctrl+C`, так как найти точное совпадение почти невозможно). Сравните показатель **Ops/sec**.

```bash
./monkey_bench en.txt random
./monkey_bench ru.txt random
./monkey_bench jp.txt random
./monkey_bench emoji.txt random
```

### Что вы увидите?
Хотя длина строки (10) одинакова, скорость (Ops/sec) будет **падать** с увеличением размера алфавита. Это связано с тем, что:
1. Увеличивается количество уникальных символов, из которых генератор выбирает случайный.
2. Многобайтовые символы (иероглифы, эмодзи) требуют больше тактов процессора для сравнения в памяти (memcmp), чем однобайтовые латинские буквы.

## ⚙️ Как это работает под капотом

### Режим Random (Генерация)
Каждый поток (Thread) создает локальный буфер на стеке размером с эталонную строку. В бесконечном цикле `while`:
1. Поток заполняет буфер случайными символами из алфавита с помощью быстрого генератора `std::mt19937`.
2. Побайтово сравнивает с эталоном.
3. При несовпадении цикл начинается сначала (память просто перезаписывается, `delete` не вызывается никогда).
4. Каждый 1 миллион итераций поток атомарно обновляет глобальный счетчик (`std::atomic`), который читается главным потоком для вывода статистики в консоль.

### Режим Brute Force (Перебор)
Строка рассматривается как число в N-ричной системе счисления (где N — размер алфавита).
1. Потоки делят пространство чисел на чанки (например, по 1 000 000).
2. Каждый поток конвертирует свое стартовое число в строку.
3. Осуществляет инкремент N-ричного числа "в столбик" с переносом разряда.
4. Это гарантирует, что два потока никогда не проверят одну и ту же комбинацию, и перебор пойдет строго последовательно: `ААА`, `ААБ`, `ААВ`...

## ⚠️ Предупреждение о бесконечности

Не ставьте в эталонный файл длинные осмысленные тексты, если запускаете в режиме `random`. 
Для строки "Привет, как дела!" (18 символов, алфавит 16 символов) математическое ожидание времени нахождения при скорости 100 млн оп/сек составляет **более 900 000 лет**.

Для проверки того, что программа реально находит совпадения и останавливается, используйте короткие строки (3-5 символов).
